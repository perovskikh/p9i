## Meta-prompt vs Prompt Sync: разграничение ролей

| Аспект | [meta-promt-adr-system-generator.md](meta-promt-adr-system-generator.md) | [promt-sync-optimization.md](../promt-sync-optimization.md) |
|--------|:---:|:---:|
| **Роль** | Конституция — **определяет** систему | Аудитор — **проверяет** соответствие |
| **Тип** | Meta | Operational (CI-ready) |
| **Содержит** | Constraints C1–C8, архитектура (Mermaid), стандартные блоки 1–5, per-prompt specs | Discovery, Coverage Matrix, Normalization, Quality Gates A–G, Sync Report |
| **Когда** | При изменении правил/архитектуры | Ежедневно/по требованию для аудита |
| **Выход** | Обновлённые промпты | Sync Report (8 секций) |

`promt-sync-optimization.md` **ссылается** на meta-prompt для определений и не дублирует их.

**Порядок запуска (стандартный поток):**
1. `meta-promt-adr-system-generator.md` — сначала (задаёт правила C1–C8)
2. `promt-sync-optimization.md` — после (проверяет соответствие)

**Исключение — эволюция системы** (обнаружена системная рассинхронизация):
1. `promt-sync-optimization.md` → Sync Report с gap
2. Обновить `meta-promt-adr-system-generator.md`
3. Снова `promt-sync-optimization.md` → применить правила

## Когда использовать какой мета-промпт

| Сценарий | Использовать |
|----------|--------------|
| Нужно изменить общие правила prompt-системы (инварианты, архитектура, синхронизация) | [meta-promt-adr-system-generator.md](meta-promt-adr-system-generator.md) |
| Нужно создать новый операционный prompt (например monitoring/security/devex) | [meta-promt-prompt-generation.md](meta-promt-prompt-generation.md) |
| Нужно обновить только логику построения очереди реализации ADR | [meta-promt-adr-implementation-planner.md](meta-promt-adr-implementation-planner.md) |
| Нужно проверить синхронизацию всей prompt-системы (CI/аудит) | [../promt-sync-optimization.md](../promt-sync-optimization.md) |

## Post-rename runbook (meta-centric)

Минимальный цикл после rename/массовых изменений prompt-системы:

1. `meta-promt-adr-system-generator.md` — зафиксировать инварианты и границы изменений.
2. `promt-sync-optimization.md` — получить Sync Report и gap.
3. `meta-promt-prompt-generation.md` — применить каскадные правки в целевые промпты.
4. `promt-quality-test.md` — проверить функциональное качество output.
5. `promt-versioning-policy.md` — подтвердить semver bump и реестр.

Обязательные команды проверки:
- `make docs-prompt-guard`
- `make docs-check-mkdocs-nav`
- `make docs-prompt-version-check`

Ожидаемые артефакты:
- `artifacts/prompt-sync/prompt-sync-latest.md`
- `artifacts/prompt-sync/prompt-sync-latest.json`

## Быстрые примеры использования

### 1) Source of Truth (архитектурные правила)

```bash
cat docs/ai-agent-prompts/meta-promptness/meta-promt-adr-system-generator.md
```

Используйте, когда нужно пересобрать/синхронизировать всю систему промптов на уровне инвариантов.

### 2) Prompt Factory (универсальный генератор)

```bash
cat docs/ai-agent-prompts/meta-promptness/meta-promt-prompt-generation.md
```

После этого дайте задачу AI в явном формате, например:  
«Используя `meta-promt-prompt-generation.md`, сгенерируй новый промпт
`promt-quality-test.md` по следующим требованиям: …»

Ожидаемый workflow мета-промпта:
`Discovery → Тип → Context7 → Сборка по скелету → Валидация → Регистрация`.

### 3) Planner Meta-Prompt (специализированный)

```bash
cat docs/ai-agent-prompts/meta-promptness/meta-promt-adr-implementation-planner.md
```

Используйте для генерации/обновления только [../promt-adr-implementation-planner.md](../promt-adr-implementation-planner.md).

## Как правильно формулировать запросы к meta-промптам

Используйте единый паттерн:

> «Используя `<meta-prompt>.md`, выполни задачу: `<scope>`.
> Ограничения: topic slug-first, dual-status, Context7 gate, READ-ONLY `docs/official_document/`.
> Ожидаемый результат: `<какие файлы и проверки>`.»

### 1) Source of Truth

> «Используя `meta-promt-adr-system-generator.md`, обнови правила prompt-системы по требованиям: …
> Синхронизируй инварианты и структуру, затем перечисли файлы для каскадного обновления.»

Ожидаемое поведение:
`Discovery → Обновление source-of-truth → Каскадные зависимости → Quality Gates`.

### 2) Prompt Factory

> «Используя `meta-promt-prompt-generation.md`, сгенерируй (или обнови) промпт `<name>.md` по требованиям: …
> Обнови реестр в `docs/ai-agent-prompts/README.md` при необходимости.»

Ожидаемое поведение:
`Discovery → Тип → Context7 → Сборка по скелету → Валидация → Регистрация`.

### 3) Planner Meta-Prompt

> «Используя `meta-promt-adr-implementation-planner.md`, обнови `promt-adr-implementation-planner.md` под требования: …
> Сохрани Critical Path, Layer 0→5, выбор следующего workflow-промпта и Context7 gate.»

Ожидаемое поведение:
`Discovery → Context7 → Обновление planner-логики → Валидация структуры → Регистрация`.

## Инварианты (обязательные для всех мета-промптов)

- ADR идентифицируются по **topic slug**, а не по номеру.
- Dual-status обязателен: `Статус решения` + `Прогресс реализации`.
- `docs/official_document/` — только READ-ONLY.
- Для архитектурных/библиотечных решений обязателен Context7 gate.
- Канонический источник схемы БД: `scripts/utils/init-saas-database.sql`.
- Legacy-контур `k8s/` не используется как активный источник.
