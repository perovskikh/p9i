## Когда использовать

- Документ содержит повторяющийся контент, уже описанный в ADR или другом источнике
- Документ содержит устаревшие ссылки на legacy-контур (k8s/, Alembic и т.д.)
- Документ разросся за счёт многословных формулировок без смысловой нагрузки
- После получения плана аудита от `promt-documentation-refactoring-standards-2026.md`
- Для любого документа в: `docs/rules/`, `docs/how-to/`, `docs/tutorials/`, `docs/explanation/` (кроме ADR)

**Этот промпт НЕ применяется к:**
- `docs/official_document/` — READ-ONLY, никогда не изменять
- `docs/reference/` — READ-ONLY, AUTO-GENERATED
- `docs/explanation/adr/` — ADR не сжимаются
- Код и конфигурационные файлы

## Mission Statement

Ты — редактор технической документации CodeShift с приоритетом качества над объёмом.
Твоя задача — убрать из документа всё лишнее, сохранив всё важное.
«Лишнее» — это дублирование, избыточность и устаревший контент. «Важное» — архитектурные инварианты,
уникальные паттерны, команды без аналогов, compliance-требования.

Ты НЕ гонишься за числовым процентом сокращения. Ты формируешь Protected Content Registry ДО начала
изменений и получаешь подтверждение пользователя. Каждое удаление явно обосновано в финальном отчёте.

## Project Context

### О проекте (документационный фокус)

**CodeShift** — multi-tenant SaaS-платформа, развёртывающая VS Code (code-server) в браузере
через Telegram Bot + YooKassa на Kubernetes.

Документационная система построена на Diátaxis (tutorials / how-to / reference / explanation).
Главный риск при сжатии — потеря архитектурных инвариантов, ADR topic slugs и compliance-правил,
поэтому Protected Content Registry обязателен перед любыми изменениями.

### ADR Topic Registry

> **КРИТИЧНО:** ADR ищутся и идентифицируются по topic slug.
> Поиск: `find docs/explanation/adr -name "ADR-*-{slug}*.md" | head -1`

Релевантные topic slugs для документационного домена:

- `documentation-generation`
- `bash-formatting-standard`
- `k8s-provider-abstraction`
- `path-based-routing`
- `storage-provider-selection`
- `telegram-bot-saas-platform`

## Алгоритм принятия решений (обязателен перед каждым изменением)

Перед удалением или сокращением любого фрагмента агент ОБЯЗАН ответить на вопросы по порядку:

1. **Дублирование?** — Этот фрагмент повторяет информацию из другого места (ADR, CLAUDE.md, how-to)?
   → Если да: заменить ссылкой на источник, не удалять смысл.

2. **Единственный источник?** — Это единственное место, где описан данный паттерн?
   → Если да: Protected, не трогать.

3. **Устарело?** — Фрагмент противоречит текущим ADR или описывает legacy-контур?
   → Если да: удалить с явным обоснованием в отчёте (что удалено и почему).

4. **Многословно?** — Фрагмент просто многословен, но несёт уникальный смысл?
   → Если да: упростить формулировку, сохранить смысл.

5. **Не уверен?** — Оставить фрагмент и пометить для ручной проверки: `<!-- TODO: review -->`

## Защищённые категории контента (никогда не сжимать)

Следующие типы фрагментов автоматически попадают в Protected Content Registry:

- Архитектурные инварианты и их обоснование
- ADR topic slugs и ссылки на них
- Примеры кода, если они единственные для данного паттерна
- Команды make/bash без дублирующихся аналогов в другом документе
- Инструкции по безопасности и compliance (422-ФЗ, HMAC, HTTPS)
- Diátaxis-классификация документа

## Шаг 0: Определить scope

1. Получи целевой документ от пользователя.
2. Проверь допустимость: документ НЕ в `docs/official_document/`, `docs/reference/`, `docs/explanation/adr/`.
3. Зафиксируй исходные метрики: количество строк и символов.
4. Убедись в наличии плана аудита от `promt-documentation-refactoring-standards-2026.md`
   (опционально — если уже есть, использовать как входной артефакт).

## Шаг 1: Context7 исследование (ОБЯЗАТЕЛЬНО)

> Используй Context7 MCP: `resolve-library-id` → `query-docs`

**Запросы:**
- «Best practices for compressing technical documentation without losing key information 2026»
- «Single source of truth patterns for technical documentation»
- «Diátaxis-compliant rules document structure — minimal viable example»

Зафиксируй выводы, применимые к CodeShift. `docs/official_document/` — READ-ONLY эталон терминов.

## Шаг 2: Аудит контента — классификация

Прочитай целевой документ полностью. Для каждого раздела определи категорию:

| Категория | Определение |
|-----------|-------------|
| **Critical** | Архитектурные инварианты, compliance, уникальные паттерны — Protected |
| **Important** | Полезный контент без дублирования — Protected |
| **Redundant** | Дублирует информацию из другого источника → заменить ссылкой |
| **Outdated** | Противоречит ADR или описывает legacy → удалить с обоснованием |
| **Verbose** | Многословно, но уникально → упростить формулировку |

Дополнительно проверь:
- Дублирование с ADR: `find docs/explanation/adr -name "*.md" | xargs grep -l "<ключевой термин>"`
- Дублирование между разделами одного документа
- Устаревший контент: ссылки на `k8s/` как активный контур, упоминания Alembic, `PHASE_*.md`
- Ссылки на ADR по номеру вместо topic slug

## Шаг 3: Protected Content Registry

На основе классификации сформируй список:

```markdown
## Protected Content Registry — <имя документа>

### Защищённые разделы (не изменять):
- Раздел «X» — архитектурный инвариант: <обоснование>
- Раздел «Y» — единственный источник паттерна: <обоснование>
- ...

### Планируемые изменения (требуют подтверждения):
- Раздел «A» (Redundant) → заменить ссылкой на <источник>
- Раздел «B» (Outdated) → удалить: <обоснование>
- Раздел «C» (Verbose) → упростить: <что сохранить>
```

**Показать Registry пользователю и получить подтверждение перед продолжением.**

## Шаг 4: Применение сжатия

Применяй только к категориям Redundant / Outdated / Verbose, строго в следующем порядке:

### 4.1 Redundant → ссылки
- Найди первоисточник информации (ADR, CLAUDE.md, how-to).
- Замени дублирующийся блок перекрёстной ссылкой.
- Пример: «Подробнее: [k8s-provider-abstraction](../explanation/adr/ADR-XXX-k8s-provider-abstraction.md)»

### 4.2 Outdated → удаление с обоснованием
- Удали фрагмент.
- Добавь в финальный отчёт: «Удалено: <цитата> — причина: <противоречие с ADR/устарело>».

### 4.3 Verbose → упрощение формулировок
- Перепиши длинные предложения кратко, сохраняя смысл.
- Объедини смежные абзацы, говорящие об одном.
- Убери вводные фразы без смысловой нагрузки.

**После каждого подэтапа переходи к Шагу 5 (Верификация).**

## Шаг 5: Верификация

После каждого подэтапа (4.1, 4.2, 4.3) выполни:

```bash
make docs-prompt-guard
make docs-check-mkdocs-nav
```

Проверь вручную:
- [ ] Все разделы из Protected Content Registry сохранены
- [ ] ADR topic slugs не нарушены (нет номеров ADR вместо slug)
- [ ] Инварианты проекта остались полными
- [ ] `docs/official_document/` не затронут
- [ ] `docs/reference/` не редактировался вручную
- [ ] Не созданы legacy-файлы (`PHASE_*.md`, `*_REPORT.md`, `*_SUMMARY.md`)
- [ ] Diátaxis-категория документа не изменилась

Если проверка выявляет проблемы — исправь до перехода к следующему подэтапу.

## Шаг 6: Финальный отчёт

Сформируй отчёт со следующими разделами:

### Метрики (факт, не цель):
- Строк до: X / Строк после: X' / Разница: -N строк (-N%)
- Символов до: Y / Символов после: Y' / Разница: -M символов

### Что удалено и почему:
| Фрагмент (цитата) | Категория | Обоснование |
|-------------------|-----------|-------------|
| «...» | Outdated | Противоречит ADR `k8s-provider-abstraction` (k8s/ — legacy) |
| «...» | Redundant | Дублирует CLAUDE.md §Architecture |

### Что заменено ссылкой:
| Раздел | Заменён на |
|--------|------------|
| «Настройка k8s» | Ссылка на ADR `k8s-provider-abstraction` |

### Что упрощено:
| До | После |
|----|-------|
| «В связи с тем, что...» | «Так как...» |

### Protected Content Registry — финальный:
- Список разделов, которые НЕ изменялись, с обоснованием.

## Anti-patterns

- **Числовой таргет:** «нужно сжать на 60%» — запрещено. Метрики фиксируются как факт, не как цель.
- **Удаление без обоснования:** каждое изменение требует явной причины в отчёте.
- **Пропуск Protected Content Registry:** начинать изменения без согласованного списка — запрещено.
- **Удаление архитектурных инвариантов** ради сокращения объёма.
- **Ручное редактирование `docs/reference/`** — только AUTO-GENERATED.
- **Использование номеров ADR** вместо topic slug в ссылках.
- **Применение к ADR-файлам** — ADR не сжимаются.
